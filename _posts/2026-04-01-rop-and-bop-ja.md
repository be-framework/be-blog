> Unixはパイプを与えた。関数型プログラミングはResultを与えた。ドメインにはどちらも要らないとしたら？

## パイプライン

Railway Oriented Programming（ROP）は、関数型プログラミングにおける最も洗練されたパターンのひとつです。その核心——ワークフローとはデータが変換を通じて流れるパイプラインであり、失敗はもう一本のレールに過ぎない。

```
Input → validate → extract → identify → save
          ↓           ↓          ↓         ↓
        Error       Error      Error     Error
```

これはUnixパイプのドメイン層への適用です。`cat | grep | sort` がテキストを変換の連鎖に流すように、ROPはドメインデータを関数の連鎖に流す——エラーハンドリングを型システムで明示しながら。

一休の飲食店向けSaaSにおける実装を見てみましょう：

```typescript
const workflow: WorkFlow = (command) =>
  ok(command)
    .andThen(validateReservationEvent)
    .andThen(extractActualVisitor)
    .asyncAndThen(identifyCustomer(findIdenticalCustomer))
    .andThen(importReservationEvent)
```

各ステップはデータを受け取り、変換したデータを送り出す関数です——あるいはエラートラックに分岐します。各段階は固有の型を持ちます：

```typescript
UnvalidatedCommand
  → SiteReservationValidated
    → VisitorExtracted
      → CustomerIdentified
        → SiteReservationEventImported
```

関数型の関心事は明確です——**データの変換**。ROPはその変換に失敗の文脈を与えました。Result型でエラーを明示し、Union型で不可能な状態を排除し、イミュータブルで破壊を防ぐ。これはDoingの最高到達点です。

## 共鳴

上の型の連鎖と、Be Frameworkの連鎖を並べてみましょう：

```
OrderInput → OrderValidated → PaymentProcessed → OrderConfirmed
```

構造的な類似は際立っています。どちらもドメインのワークフローを型付けされた段階の列として表現しています。どちらもUnion型で存在しうるものだけを制約します。どちらもイミュータブルです。どちらも各遷移を明示します。

これは偶然ではありません。どちらも同じ問題——制約のない状態の混沌——に応答しています。関数型アプローチと存在指向アプローチは、異なる出発点から似た構造に辿り着きました。

## 主語

似ているのは構造です。違うのは文法です。

```typescript
// ROP: システムがデータを変換する
const archived = archiveCustomer(customer)
```

```php
// BOP: ドメインが自ら変容する
$becoming(new PatientArrival($vitals))
```

ROPでは `extractActualVisitor(command)` ——システムが来店者を抽出します。パイプラインがデータを処理します。関数はオブジェクト*に*適用されます。主語はシステム、三人称です。

BOPでは、`PatientArrival` がトリアージプロトコルと出会い、`EmergencyCase` に*なります*。主語はドメイン自身、一人称です。抽出する者はいません。処理する者もいません。存在が変容します。

この違いはアーキテクチャにまで及びます。ROPのワークフローはオーケストラの指揮者です——総譜を知り、各楽器を順番に導きます。BOPでは各存在が `#[Be]` を宣言し——何に成れるかを——連鎖が自己組織化します。これはコレオグラフィーです。各ダンサーは自分のパートだけを知っている。けれど全体は調和します。

## 問い

ROPが問うのは：**この変換は成功したか、失敗したか？**

これはHOWの問いです。エラーをどう扱うか。失敗をどう伝播させるか。失敗しうる関数をどう合成するか。ROPはこれらの問いに美しく答えます——おそらくこれ以上ないほどに。

BOPが問うのは：**そもそもこれは存在できるか？**

これはWHETHERの問いです。`Email` はバリデーションされて `Result<Email, Error>` を返すのではありません。存在するか、しないか。エラートラックがないのは、トラックそのものがないからです。意味変数は、不正な値が捕捉されて分岐する世界ではなく、不正な値がそもそも存在できない世界を定義します。

この問いの転換から導出されるテクニック——自己証明、自己決定、コレオグラフィー、イミュータブル——は個別の機能ではありません。すべてが一つの視点の変化から導かれています：**ドメインを一人称で見る**。

ROPは答えを精緻化します。BOPは問いを変えます。

## 置き換えではない

ROPは間違っていません。Doingパラダイムの最高到達点のひとつでしょう——データ変換を安全に、明示的に、合成可能にした。一休での2年間の本番運用が、その実践的な強さを裏付けています。バグは減り、エラーハンドリングは改善され、型安全性は堅牢です。

BOPはこれを置き換えるものではありません。変換について考えることをやめて、存在について考え始めたら何が起こるかを問うています。両者は同じ風景を異なる高さから見ています。一方からはレールとポイントが見える。もう一方からは、列車がレールと分かれていなかったことが見えます。

視点は積み重なるものです。一方を捨てなければ他方に辿り着けないわけではありません。
